## Part 1: Monitoring and Observability

While the terms are often used interchangeably, they represent two different stages of system visibility. **Monitoring** tells you _when_ something is wrong, while **Observability** gives you the internal context to understand _why_ it is wrong.

### The Three Pillars of Observability (M.E.L.T.)

To achieve true system observability, you must collect and analyze three distinct types of telemetry data:

- **Metrics (Numerical Data):** Aggregatable, numeric data points measured over intervals of time (e.g., CPU utilization percentage, memory usage, request counts per second, error rates). They are lightweight to store and ideal for triggering real-time alerts.
    
- **Logs (Textual Events):** Immutable, timestamped text records of discrete events that happened within an application or system (e.g., a stack trace, a failed login attempt, an database connection timeout). Logs provide critical textual context during debugging.
    
- **Traces (Request Journeys):** A representation of a single request’s end-to-end journey as it moves through a distributed system or microservices architecture. It maps the latency and health of every individual hop (span) across network boundaries.
    

### Prometheus and Grafana Ecosystem

This is the industry-standard open-source combination for handling metrics and visualization.

- **Prometheus (The Data Gatherer):**
    
    - **Pull-Based Architecture:** Unlike traditional monitoring tools that wait for servers to send data, Prometheus actively reaches out to target servers via HTTP endpoints (usually `/metrics`) at regular intervals to scrape telemetry data.
        
    - **Time-Series Database (TSDB):** It stores all data as a stream of timestamped values identified by metric names and key-value pairs (labels).
        
    - **PromQL:** Uses its own powerful functional query language to compute and aggregate metric data on the fly.
        
- **Grafana (The Visualizer):**
    
    - Grafana does not store data itself. Instead, it connects to data sources like Prometheus, Elasticsearch, or AWS CloudWatch.
        
    - It translates complex PromQL queries into beautiful, real-time, interactive **dashboards** filled with graphs, heatmaps, and gauges so operations teams can view system health at a glance.
        

## Part 2: Cloud Security Concepts

Security in the cloud shifts from physical perimeter defense (fences and locked rooms) to software-defined identity and access perimeters.

### The Shared Responsibility Model

When operating in the cloud, security is a shared effort between the cloud service provider (CSP) and the customer. The exact boundary line depends on whether you are using IaaS, PaaS, or SaaS, but the general breakdown is:

- **Security OF the Cloud (Provider's Responsibility):** The CSP secures the physical infrastructure, global network, data centers, host operating systems, and virtualization software layers running the services.
    
- **Security IN the Cloud (Customer's Responsibility):** You are responsible for securing whatever you put _inside_ the cloud. This includes customer data, application code, identity management, network traffic access controls (firewalls), and operating system patching (on VMs).
    

### Identity and Access Management (IAM)

IAM is the security framework that ensures the right individuals and systems have access to the right cloud resources under the right conditions. It centers on three core principles:

1. **Authentication (AuthN):** Verifying _who_ you are (e.g., username/password, Multi-Factor Authentication (MFA)).
    
2. **Authorization (AuthZ):** Verifying _what_ you have permission to do once authenticated, managed through policies and roles.
    
3. **Principle of Least Privilege (PoLP):** The fundamental security rule stating that users, processes, and service accounts must be granted _only_ the absolute minimum privileges necessary to complete their specific tasks, and nothing more.
    

## Part 3: DevSecOps and Pipeline Security

**DevSecOps** is the practice of integrating security testing and compliance protocols directly into every phase of the traditional DevOps delivery pipeline, shifting security "left" (earlier in the lifecycle) rather than treating it as an afterthought.

```
[Plan] ➔ [Code] ➔ [Build] ➔ [Test] ➔ [Deploy] ➔ [Monitor]
   ▲        ▲        ▲        ▲         ▲          ▲
   └────────┴────────┴─── Security Integrated ─────┴───┘
```

### DevSecOps Tools and Methods

Security automation tools are injected into different phases of the CI/CD pipeline to catch vulnerabilities automatically:

- **SAST (Static Application Security Testing):** Scans raw source code before it is compiled or executed to find security flaws, hardcoded credentials, and formatting weaknesses (e.g., SonarQube, Checkmarx).
    
- **SCA (Software Composition Analysis):** Analyzes open-source third-party dependencies and libraries used by your application to flag known vulnerabilities (CVEs) and licensing non-compliance (e.g., Snyk, OWASP Dependency-Check).
    
- **DAST (Dynamic Application Security Testing):** Attacks a running instance of the application in a staging environment from the outside to discover runtime vulnerabilities like SQL injections or Cross-Site Scripting (XSS) flaws (e.g., OWASP ZAP).
    

### Securing CI/CD Pipelines and Cloud Infrastructure

- **Secrets Management:** Ensure api keys, cloud credentials, and database passwords are encrypted and injected dynamically at runtime using secure vaults (like HashiCorp Vault or AWS Secrets Manager) instead of being stored in git repositories.
    
- **Network Segmentation:** Implementing strict network configurations—such as security groups, Virtual Private Clouds (VPCs), and Private Subnets—to isolate database clusters and backend APIs from the public internet.
    
- **Immutable Infrastructure:** Deploying updates by tearing down old servers and launching entirely new ones pre-configured from scratch, minimizing the risk of "configuration drift" or lingering security malware on live servers.
